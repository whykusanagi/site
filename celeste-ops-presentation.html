The Problem02 / 13
Your agents have nowhere durable to work.
Your agents draft specs, write code, triage issues, and queue posts. They need a durable place to put that work so you can review it, and so the next agent reads it back.
Specs rot or vanish
Commit a plan to the repo and you ship half-formed thinking in your git history, stale by morning. Leave it out and it dies the second an agent's context resets. The next subagent re-derives the same decisions.
Coordination lives elsewhere
Slack, Notion, and GitHub each solve a slice, behind auth boundaries, rate limits, and a network round-trip for every read. The work runs on your laptop. Routing the loop through someone else's servers is backwards.
Five drifting copies
Each agent keeps its own private list. By Thursday they disagree about what's done, what's blocked, and what comes next.
Localhost is not a wall
A local app that opens an HTTP port answers any web page you visit. "It came from 127.0.0.1" decides nothing about trust.

The Differentiator05 / 13
An approve button other processes wait on.
A notes app doesn't have one. The loop that makes a shared backlog work:
01 · agent
Files a spec over MCP, flagged in-review
→
02 · you
Hit Approve in the Notes tab, in plain English
→
03 · system
The blocked task clears and slides into the Inbox
→
04 · next agent
Polls the timestamp and picks it up
↺ repeats · new session · new repo · new model
agent A → document_create(folder:"specs", review_status:"in-review")
you → click Approve → document_update → approved
       blocked task clears → Inbox
agent B → tasks_unblocked() → [ the freed task ]
The backlog doubles as a message board between agents. Edit an approved spec and CelesteOps drops it back to modified, so the next agent re-reads before it acts on a stale plan.
Why It Matters06 / 13
What a vault and a SaaS board leave on the table
Specs out of git, still queryable
Plans live as documents tagged by metadata: repo, folder, review_status. The git tree stays clean. The plan stays durable across context resets.
No network tax
Every read and write is a local SQLite call. No auth dance, no rate limit, no round-trip. The coordination layer runs where the work runs.
You hold the approve button
Agents can't ship themselves. Every spec, every credential, every prototype waits on a first-party click an agent can't make.
Yours, signed, encrypted
A PGP-signed build you verify before you trust it, and AES-256-GCM backups behind a macOS Keychain key. Off-site copies stay encrypted and stay yours.

A site with no token gets a 401, even if it guesses the port.
Encrypted at rest. R2 keys and the backup password sit behind a macOS Keychain key; the API returns configured: true, never the value.
Host-welded broker. Agents reach one fixed host, SSRF-guarded against localhost and private IPs; only MCP-enabled credentials are discoverable.
Verifiable releases. Every build is PGP-signed. You run gpg --verify before you trust it.
Sandboxed prototypes. Agent-authored HTML renders in a null-origin iframe with a locked CSP, and waits for your approval to run.
No self-approval. The app's own UI uses an ephemeral in-memory token; the pairing code is the consent, so an agent can never enroll or approve itself.

Monetization10 / 13
